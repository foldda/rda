Manifest No. RDA‑02 Data+Schema RDA encoding demonstration

Same content, different packages

This example demonstrates encoding data of a freight yard ships parcels using data+schema RDAs.

On the left, the data-RDA box carries only values — no field names travel with every record, and a separate manual, a schema-RDA object, carries the meaning (the meta-data). The output RDA strings are compared to the JSON-encoded string that carries the same information on the right, where names and values travel together in every record.

Pick a scenario to see exactly what has to change from the base shipment, in red.

The manual is paid once per integration, not once per shipment. As record count grows, its cost is amortized toward zero, while JSON keeps repeating every field name in every record. This is the number that actually matters for a real integration, not the single-shipment table above it.
How to read this: the data object never carries field names — only the manual does, sent once as a sibling RDA object. The manual itself carries no header row either: position 0 is always the first entry, by convention, the same way the data has no title row. That's why a pure rename costs the data object nothing. The manual isn't limited to a flat index→name list — as the "Destination added" case shows, an entry can itself declare sub-attribute names, using the same recursive delimiter rule as any RDA data. Adding a trailing field only touches the rows that actually have it — since it's the last column, rows without it simply end early, no placeholder needed. Removing a field needs no change to the manual at all. This is a simplified, illustrative encoding built from the two examples in the RDA README, not the full project spec.
